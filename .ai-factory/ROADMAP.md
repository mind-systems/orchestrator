# Project Roadmap

> Orchestrator ‚Äî AI-driven multi-agent pipeline that autonomously plans, implements, and reviews code milestones.

## Milestones

- [x] **Orchestrator state file** ‚Äî Add `orchestrator-state.json` to track inter-run state. Implement `_load_state` / `_save_state` helpers. Record review file names created during implement phase so they can be selectively deleted before the review phase, instead of wiping all reviews indiscriminately.

- [x] **Preparatory refactor** ‚Äî Two small changes before refactor mode. (1) Replace hardcoded `MAX_REVIEW_ITERATIONS = 3` with env var `ORCHESTRATOR_MAX_REVIEW_ITERATIONS` (default 3); add `ORCHESTRATOR_MAX_REFACTOR_ITERATIONS` (default 2); pass limits as parameters to `process_milestone()` instead of reading the global. (2) Add `PipelineStopError(message)` to `agents.py`; catch it in `cli()` alongside `RateLimitError` ‚Äî print message and exit 0.

- [x] **RefactorPlanner agent class** ‚Äî Add `RefactorPlanner` to `agents.py` using `refactor-planner.md` as system prompt. Method `audit_and_plan(milestone_title, milestone_description, plan_path)` for the first iteration. Method `verify(plan_path, review_path)` for subsequent iterations ‚Äî writes findings to review file, returns `True` if file ends with `REVIEW_PASS`, `False` otherwise. Same session across calls via `--resume`.

- [x] **process_refactor_milestone function** ‚Äî Add `process_refactor_milestone(project_dir, milestone, milestone_index)` to `main.py`. Chain: `RefactorPlanner.audit_and_plan` ‚Üí `Implementer.implement` ‚Üí `RefactorPlanner.verify`. On `REVIEW_PASS` mark done and git commit. If max iterations reached without `REVIEW_PASS` ‚Äî raise `PipelineStopError` with the last review file path and its contents. Uses `ORCHESTRATOR_MAX_REFACTOR_ITERATIONS` limit.

- [x] **refactor CLI command** ‚Äî Add `refactor` subcommand to `cli()` in `main.py`. Reads ROADMAP.md, filters pending milestones, runs `process_refactor_milestone` for each. Wraps in `_with_caffeinate`. Handles `PipelineStopError` by printing findings and stopping ‚Äî does not continue to next milestone.

- [x] **Unified iteration limit** ‚Äî Replace all separate iteration limit constants and env vars (`ORCHESTRATOR_MAX_REVIEW_ITERATIONS`, `ORCHESTRATOR_MAX_REFACTOR_ITERATIONS`, etc.) with a single `ORCHESTRATOR_MAX_ITERATIONS` env var (default 3). All flows ‚Äî implement review, plan review, refactor ‚Äî use this one value. Update all call sites in `main.py` that pass iteration limits.

- [x] **Plan review cycle** ‚Äî Give the plan phase up to `ORCHESTRATOR_MAX_ITERATIONS` attempts instead of a single pass. PlanReviewer writes its output to `.ai-factory/plan-reviews/{seq}-{slug}-plan-review-{n}.md` and ends the file with `PLAN_REVIEW_PASS` if the plan is solid. Planner reads the review file on the next attempt (not raw text output). If `PLAN_REVIEW_PASS` is not reached by the last iteration ‚Äî raise `PipelineStopError` with the last plan-review file path and contents.

- [x] **Graceful stop in review loop** ‚Äî `_implement_loop` and `_refactor_loop` check `state.stop_requested` before each milestone, but the review loop inside `run_review` does not. Extract a shared `_run_loop(items, process_fn)` helper that checks `state.stop_requested` before each item and prints `>>> Stop requested ‚Äî halting.` if set. Replace the `for` loops in `_implement_loop`, `_refactor_loop`, and the inner `loop()` in `run_review` with calls to `_run_loop`. Also add a `state.stop_requested` check in `run_implement_review` between the implement phase and the review phase ‚Äî so that Ctrl+C during implement prevents the review phase from starting.

- [x] **Roadmap context in agent prompts** ‚Äî `Milestone` already has `line_number` (0-based). Pass `roadmap_path` and `milestone.line_number` into `PlannerReviewer.plan()`, `RefactorPlanner.audit_and_plan()`, and `Implementer.implement()`. In each prompt, add a line before the milestone title: `"Roadmap: {roadmap_path} (line {milestone.line_number + 1})\n"`. The agent then has the option to read the full roadmap for context ‚Äî what's already done, what comes next ‚Äî without being forced to.

- [x] **Fix REVIEW_PASS gate in reviewer prompt** ‚Äî The code reviewer bypasses the "zero issues" rule by inventing non-standard section names (`## Bugs`, `## Issues`) instead of `## Critical Issues`. The prompt must make the gate content-based, not section-name-based. In `orchestrator/prompts/reviewer.md`, replace the current REVIEW_PASS rules with: "Write `REVIEW_PASS` only if you have no findings at all ‚Äî every findings section you wrote is empty. If you wrote even one bug, issue, or problem under any heading, do not write `REVIEW_PASS`." Also remove the `### Suggestions` section from the output format ‚Äî suggestions are findings too and must block PASS.

- [x] **Resume from mid-milestone failure in implement mode** ‚Äî `process_milestone()` currently always starts from step 1 (plan). Add a `_detect_milestone_step()` helper that inspects existing artifacts and returns the step to resume from. Logic: if no plan file ‚Üí `plan`; if no plan-reviews ‚Üí `plan_review` (attempt 1); if latest plan-review lacks `PLAN_REVIEW_PASS` ‚Üí `plan` (revision, attempt N+1); if `git diff HEAD` is empty ‚Üí `implement`; if no reviews dir or no review files for this slug ‚Üí `review` (iteration 1); if latest review lacks `REVIEW_PASS` ‚Üí `implement` (next iteration). Call `_detect_milestone_step()` at the top of `process_milestone()` and skip already-completed steps. Pass the detected `attempt`/`iteration` counter so the file naming stays correct.

- [x] **Plan review in refactor mode** ‚Äî Add `PlanReviewer` phase to `process_refactor_milestone()` after `audit_and_plan()`, identical to how it works in `process_milestone()`. PlanReviewer checks that the refactor plan doesn't break existing functionality ‚Äî not whether the audit findings are correct, but whether the proposed changes are safe. Uses the same `plan-reviews/` directory and `PLAN_REVIEW_PASS` signal. If plan review fails after `ORCHESTRATOR_MAX_ITERATIONS` attempts ‚Äî raise `PipelineStopError`.

- [x] **Resume from mid-milestone failure in refactor mode** ‚Äî Same idea for `process_refactor_milestone()`, but accounting for the plan review phase added above. Detection logic: if no plan file ‚Üí `audit_and_plan`; if no plan-reviews ‚Üí `plan_review` (attempt 1); if latest plan-review lacks `PLAN_REVIEW_PASS` ‚Üí `audit_and_plan` (revision, attempt N+1); if `git diff HEAD` is empty ‚Üí `implement`; if no review files for this slug ‚Üí `verify` (iteration 1); if latest review lacks `REVIEW_PASS` ‚Üí `implement` (next iteration). Skip completed steps, pass correct iteration counter.

- [x] **Fix seq mismatch in resume detection** ‚Äî `_next_number()` scans `plans/` for the highest numeric prefix and returns `max + 1`. If a previous interrupted run already wrote a new plan file (e.g. `49-nfb-tab-calibration.md`), the next run gets `seq=50`, then `_detect_milestone_step()` looks for `50-nfb-tab-calibration-plan-review-*.md`, finds nothing, and falls back to `step="plan"` ‚Äî ignoring the existing `48-*` artifacts. The same seq mismatch affects all artifact lookups: `plan-reviews/`, `reviews/`, and `patches/`. Fix: in `_detect_milestone_step()`, before using the passed `plan_path`, scan `plans/` for any existing file matching `*-{slug}.md` (glob by slug, ignore seq). If found, derive the canonical seq from that file and use it for all artifact directory lookups (`plan-reviews/`, `reviews/`, `patches/`). If multiple matches exist (interrupted run left duplicates), use the one with the lowest seq ‚Äî that's the original. The caller-provided `plan_path` (with the new seq) should only be used if no existing slug match is found. This fix covers both `process_milestone()` and `process_refactor_milestone()` since both call `_detect_milestone_step()`. Additionally, when a canonical seq is found, `_detect_milestone_step()` must return the canonical `plan_path` (pointing to the original plan file) alongside `(step, counter)` ‚Äî callers must use this returned path instead of their locally constructed one for all subsequent agent calls (`planner_reviewer.plan()`, `implementer.implement()`, `planner_reviewer.review()`, etc.).

- [x] **Log session_id on new session start** ‚Äî After each first `_run_claude()` call that creates a new session, print the session_id so it's visible in the orchestrator log. In `PlannerReviewer.plan()`, `PlanReviewer.review_plan()`, `Implementer.implement()`, and `RefactorPlanner.audit_and_plan()`: after the `_run_claude()` call that assigns `self.session_id` for the first time (i.e. when `self.session_id` was `None` before the call), add `print(f"  [session: {self.session_id}]")`. For `PlanReviewer` which has no persistent `session_id`, log unconditionally after `_run_claude()`.

- [x] **Elapsed time in closed milestones** ‚Äî When marking a milestone done in `ROADMAP.md`, append the elapsed time to the end of the line (e.g. `[12m 34s]`). Time is calculated before `mark_done()` and passed as `elapsed_secs` parameter.

- [x] **Stop on review iteration limit** ‚Äî `process_milestone()` was silently committing when `max_iterations` was reached without `REVIEW_PASS`. Replace the `WARNING` print with a `PipelineStopError` ‚Äî consistent with `process_refactor_milestone()` behavior.

- [x] **Breakpoint in ROADMAP.md** ‚Äî Add support for `---STOP---` marker in ROADMAP.md. In `roadmap.py`, `parse_roadmap()` currently reads all lines and collects milestones. Add a check: if a line strips to `---STOP---`, stop collecting milestones at that point ‚Äî milestones after the marker are not returned. In `_implement_loop()` and `_refactor_loop()`, update the log line to indicate when a breakpoint was hit ‚Äî but only when the marker actually cut off pending milestones. `parse_roadmap()` must return whether a `---STOP---` marker was encountered and how many milestones were left after it. If the marker was present AND there are milestones after it that were not returned, print `Found N pending milestones out of M total (stopped at breakpoint ‚Äî X milestones after marker not queued).` If the marker is present but nothing follows it (e.g. it's at the end of the file), treat it as if it weren't there ‚Äî no special message. `review` mode reads `plans/` directly and is not affected by this marker.

- [x] **Replace LLM review with real test runner in `test` mode** ‚Äî Currently `run_test()` in `orchestrator/main.py` calls `_implement_loop(project_dir, max_iterations, "test-planner", "ROADMAP_TESTS.md")` ‚Äî which is identical to the implement pipeline: Plan ‚Üí PlanReview ‚Üí Implement ‚Üí **LLM CodeReview**. The LLM reviewer reads the written test file and judges whether it looks correct, but cannot actually run the tests. A test that compiles but tests the wrong invariant will get `REVIEW_PASS` from the LLM. The fix: add a `process_test_milestone()` function in `main.py` that replaces the LLM review step with a real shell test run. The new pipeline is: Plan (test-planner prompt) ‚Üí PlanReview ‚Üí Implement ‚Üí **TestRunner** ‚Üí mark done + commit. `TestRunner` is a new class in `agents.py` ‚Äî not an LLM agent, just a shell executor. It reads the test command from the plan file (the test-planner prompt already writes `## Test Command` with the exact command), runs it via `subprocess.run()` in `project_dir`, captures stdout+stderr, writes the full output to `.ai-factory/test-runs/{seq}-{slug}-test-{n}.txt`, and returns `True` if exit code is 0. The Implementer on the next iteration reads this output file alongside the plan (same pattern it uses for review files today ‚Äî pass `test_run_path` the same way `review_path` is passed). Add a `test-runs/` directory alongside `plans/`, `reviews/`, `plan-reviews/`. Add `_test_loop()` in `main.py` that mirrors `_implement_loop()` but calls `process_test_milestone()` instead of `process_milestone()`. Update `run_test()` to call `_test_loop()` instead of `_implement_loop()`. Add `resume` support in `_detect_test_milestone_step()` ‚Äî same logic as `_detect_milestone_step()` but using `test-runs/` instead of `reviews/` for the run files. Do NOT change `process_milestone()`, `_implement_loop()`, or any other existing pipeline. Do NOT change the `test-planner.md` prompt ‚Äî it already writes the test command in the correct format.

- [x] **Persist agent session IDs in plan file** ‚Äî `PlannerReviewer` and `Implementer` lose `self.session_id` on crash/restart; on resume, `_detect_milestone_step()` correctly finds the right step but both agents start fresh sessions, losing planner context and implementer working state. Fix: store session IDs in the plan file as an HTML comment block at the end (`<!-- orchestrator-sessions\nplanner: <id>\nimplementer: <id>\n-->`). Full spec and helper code in `notes/03`. In `agents.py`: add `_read_sessions(plan_path)` and `_write_session(plan_path, role, session_id)` module-level helpers (using `re.compile` on the comment block); call `_write_session` after each `_run_claude` in `PlannerReviewer.plan()`, `PlannerReviewer.review()`, and `Implementer.implement()` ‚Äî `plan_path` is already a param of all three. In `main.py`: add `_read_sessions` to the import; in `process_milestone()` and `process_test_milestone()` after agents are created and `plan_path` is canonical, read sessions and assign `planner_reviewer.session_id` and `implementer.session_id`. Scope: implement and test flows only ‚Äî not refactor. Depends on the test-runner milestone for `process_test_milestone()` to exist. [6m 52s]
