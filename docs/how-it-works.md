# Как это работает

## Агентный конвейер

Каждый milestone проходит через две фазы, каждая из которых ограничена одним и тем же лимитом итераций.

Сначала идёт план-фаза: PlannerReviewer исследует кодовую базу и пишет план. PlanReviewer проверяет его и пишет результат в `.ai-factory/plan-reviews/` — либо с `PLAN_REVIEW_PASS` в конце, либо с описанием проблем. Если план не прошёл — Planner читает файл ревью и переписывает план. Цикл повторяется до `max_iterations` раз (задаётся в `orchestrator.json`).

Когда план утверждён, начинается implement-фаза: Implementer пишет код, PlannerReviewer ревьюит изменения и пишет результат в `.ai-factory/reviews/`. Если проблемы найдены — путь к файлу ревью передаётся Implementer'у явным параметром, он читает его и применяет правки, ревью повторяется. Лимит тот же — `max_iterations`. Исчерпание лимита любой из фаз без PASS-подписи — это отказ milestone (см. [Отказы и остановки](failures-and-halts.md)).

## Сессии агентов

Planner и Reviewer — один агент (`PlannerReviewer`), работающий в одной Claude-сессии через `--resume`. Это значит, что Reviewer знает весь контекст планирования — он понимает не только что было написано, но и почему. Implementer тоже держит сессию открытой между итерациями fix-цикла, накапливая контекст правок.

PlanReviewer — отдельный агент с чистой сессией на каждую попытку. Это намеренно: он смотрит на план без предвзятости автора, как независимый ревьюер.

При каждом запуске агента в лог выводится session_id созданной сессии — он виден сразу после строки `--- Claude agent (...) ---`. По этому идентификатору можно найти файл сессии в `~/.claude/projects/` для диагностики.

## Resume после прерывания

Если оркестратор прервался в середине milestone (таймаут, Ctrl+C, сбой сети), при следующем запуске он автоматически определяет с какого шага продолжить — без ручного вмешательства.

Каждый раз когда фаза завершается, оркестратор записывает её имя в JSON-файл рядом с планом, с тем же стемом: `.ai-factory/plans/{NN}-{slug}.json` (для именованного роадмапа — `.ai-factory/plans/{slug-роадмапа}/{NN}-{slug}.json`, см. ниже). Там хранятся `step` (текущая фаза), session ID планировщика и имплементера (для resume через `--resume`), и накопленное время. При следующем запуске оркестратор читает `step` и сразу переходит к нужной фазе, не повторяя уже выполненное.

Возможные значения `step`: happy-path `planned` → `plan_reviewed` → `implemented`, плюс промежуточные `plan_review_failed:N` и `review_failed:N` (в test-режиме — `test_run_failed:N`) — они записываются после провального раунда и указывают, с какой попытки продолжать. Если `step` ссылается на файл которого нет на диске (например, после `/task-rescue` удалил план-ревью), значение игнорируется и оркестратор переходит к эвристическому детекту по артефактам на диске.

Выбор плана для resume опирается на файловый протокол артефактов: закоммиченный и чистый (tracked+clean) файл плана считается частью уже завершённого milestone и пропускается — адаптируется только незакоммиченный (in-flight) план; если разработчик вручную закоммитил артефакты недоделанного milestone, это ведёт к повторному планированию, а не к ложному завершению.

## Фазы роадмапа и сессии планировщика

Оркестратор группирует milestone-ы по фазам — любой `##` или `###` заголовок в роадмапе является границей фазы. Внутри одной фазы `PlannerReviewer` держит одну Claude-сессию: каждый следующий milestone продолжает её через `--resume`, не начиная заново.

Это означает, что к третьему milestone фазы планировщик уже знает `ARCHITECTURE.md` и контекст предыдущих планов — не тратит токены на повторное чтение. На границе фазы сессия сбрасывается.

Фазовая сессия живёт только в памяти текущего процесса. Если оркестратор перезапустить между milestone-ами, следующий milestone стартует с чистой сессией — без потери корректности.

Поле `enable_phase_sessions: false` в конфиге отключает carry-forward: каждый milestone начинает со свежей сессии PlannerReviewer даже внутри одной фазы. Это позволяет сравнить расход токенов — прогнать одну фазу дважды (с флагом и без) и сравнить строки `[usage: ...]` в логе.

## Лимит использования сессии

Перед каждым milestone оркестратор запускает `claude /usage` и проверяет два независимых порога: сессионный (`usage_threshold_5h`, по умолчанию 90) и недельный по всем моделям (`usage_threshold_weekly`, по умолчанию 95). Превышение любого из них останавливает прогон до начала нового milestone.

В лог выводится строка вида `[usage: session 26% · week 52%]` — по ней видна динамика накопления на каждый milestone. Если `claude /usage` недоступен или не удалось распарсить вывод — предупреждение в лог, прогон продолжается.

## Файловый протокол

Агенты не общаются напрямую — только через файлы в `.ai-factory/` целевого проекта. Planner пишет план, Implementer его читает. PlanReviewer пишет ревью плана в `plan-reviews/`, Planner читает его при следующей попытке. Reviewer пишет ревью кода в `reviews/`, путь к этому файлу передаётся Implementer'у явным параметром на следующей итерации. Оркестратор координирует порядок вызовов и следит за сигналами (`PLAN_REVIEW_PASS`, `REVIEW_PASS`).

Для дефолтной пары `ROADMAP.md`/`ROADMAP_TESTS.md` артефактные директории (`plans/`, `plan-reviews/`, `reviews/`, `test-runs/`) остаются плоскими — байт-в-байт как раньше. Для любого другого (именованного) роадмапа артефакты уходят в поддиректорию, ключом для которой служит стем файла роадмапа: например, для `roadmaps/john-doe.md` — `plans/john-doe/`, `reviews/john-doe/`. Тестовый сиблинг именованной пары делит ту же поддиректорию с основным роадмапом: `roadmaps/john-doe-tests.md` ключуется тем же стемом `john-doe`, что и `roadmaps/john-doe.md`, — оба, implement- и test-mode артефакты именованной пары, делят одну ось нумерации `plans/john-doe/`, точно как дефолтная пара делит плоские директории. Это разводит нумерацию `{NN}` разных пользователей по разным осям, чтобы слияние веток не порождало коллизий и конфликтов при merge.

## Сигналы завершения

Все агенты используют один и тот же протокол: пишут результат в файл, последней строкой ставят сигнал если всё хорошо — или оставляют файл без сигнала если проблемы остались. PlanReviewer пишет `PLAN_REVIEW_PASS`, Reviewer пишет `REVIEW_PASS`. Оркестратор читает файл, считает итерации и решает — продолжать, или остановиться и показать последнее ревью пользователю.

Файл ревью может содержать секцию `## Deferred observations` перед сигнальной строкой — это неблокирующий канал для проверенных, но осознанно не заблокированных наблюдений вне рамок milestone. Наличие этой секции не влияет на PASS/FAIL: оркестратор по-прежнему определяет результат только по последней строке файла.
