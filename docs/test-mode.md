# Режим тестирования

## Зачем отдельный режим

Написание тестов — специализированная задача: Planner должен читать исходный код, выводить из него конкретные случаи поведения и формулировать их в виде `should … when …` тест-кейсов. Обычный плановщик это не делает — он планирует реализацию фич. Режим `test` подключает специализированный промпт планировщика под эту задачу, оставляя весь остальной пайплайн без изменений.

Тест-задачи хранятся в отдельном файле `ROADMAP_TESTS.md` — чтобы не засорять основной роадмап.

## Как это работает

```
PlannerReviewer.plan()      ← использует test-planner промпт
  └─► PlanReviewer.review_plan()  ×N
        └─► Implementer.implement()
              └─► TestRunner.run()  ×N   ← реальный запуск тестов, не LLM
                    └─► mark_done() + git commit
```

Ключевое отличие от `implement` — финальный шаг. Вместо LLM-ревьюера, который читает тест-файл "на глаз", запускается реальная тест-команда. `TestRunner` читает секцию `## Test Command` из план-файла (test-planner всегда её пишет), запускает команду через shell, захватывает stdout+stderr и exit code. Если exit code 0 — milestone завершён. Если нет — вывод тестов попадает в `patches/` и Implementer читает его на следующей итерации как обычный патч.

Это устраняет класс ошибок, который LLM не видит: тест компилируется, выглядит правдоподобно, но падает при запуске из-за неверного мока или неправильного инварианта.

PlannerReviewer в режиме `test` использует специализированный промпт: читает исходный код целевых файлов целиком, находит существующие `*.spec.*` / `*.test.*` паттерны и пишет план где каждая задача — один `describe`-блок с именованными тест-кейсами. Обязательное поле плана — `## Test Command` с точной командой запуска.

## Формат milestone-ов

Milestone-ы для `test` режима должны явно называть что тестировать:

```
- [ ] **Tests: TradeAggregator** — Написать unit-тесты для `src/candles/trade-aggregator.ts`.
  Target file: `src/candles/trade-aggregator.spec.ts`.
```

Чем конкретнее указаны целевые файлы и классы, тем точнее план.

## Запуск

```bash
uv run orchestrator test /path/to/project
```

Оркестратор читает `.ai-factory/ROADMAP_TESTS.md` целевого проекта. Основной `ROADMAP.md` не затрагивается. После каждого milestone делается git commit и задача помечается выполненной в `ROADMAP_TESTS.md`.
