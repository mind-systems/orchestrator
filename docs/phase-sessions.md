# Фазовые сессии и расход токенов

`enable_phase_sessions` управляет тем, переносится ли сессия `PlannerReviewer` между task-ами внутри одной фазы роадмапа. По умолчанию — `false`. Этот документ объясняет, почему: что было измерено и какой из этого вывод.

## Вопрос

Сквозная сессия задумана как межзадачная память: к третьему task планировщик помнит контекст предыдущих планов и не перечитывает его заново. Возникли две гипотезы о стоимости:

1. Перенос сессии экономит токены (меньше повторного чтения контекста).
2. Если экономии нет, то проблема — в дороговизне `--resume`: каждый возврат в сессию перечитывает растущий транскрипт. Тогда решением было бы держать один процесс агента живым через всю фазу и слать промпты в него, а не перезапускать `claude --resume` на каждый ход.

Обе проверены измерением.

## Измерение: resume против живого процесса

Контролируемый тест с байт-идентичными промптами в двух режимах вызова одной сессии:

- **resume** — процесс умирает между ходами, каждый ход поднимается заново через `claude --resume <sid>`;
- **live** — один процесс `claude -p --input-format stream-json` живёт между ходами, промпты идут ему в stdin.

Каждый режим прогнан с паузой 0 и с паузой 6.5 минут между ходами (имитация работы других агентов в пайплайне). Метрика — разбивка input-токенов второго хода: `cache_read` (чтение из кэша, ~10% цены) против `cache_creation` (запись в кэш, ~125%).

| Режим | input | cache_creation | cache_read |
|---|---|---|---|
| resume, без паузы | 10 | 319 | 37161 |
| live, без паузы | 10 | 340 | 37202 |
| resume, пауза 6.5 мин | 10 | 80 | 37202 |
| live, пауза 6.5 мин | 10 | 335 | 37202 |

Вывод: `cache_read` идентичен во всех четырёх ячейках, включая `resume` после паузы. То есть:

- Prompt-кэш Claude Code переживает паузу в минуты (используется расширенный TTL, не дефолтные 5 минут), поэтому `--resume` полностью переиспользует закэшированный префикс.
- **Живой процесс не экономит ни одного токена** по сравнению с `--resume`. API без состояния: полная история отправляется на каждом ходу в обоих режимах, а удешевляет повторную отправку только серверный кэш — он привязан к wall-clock TTL, а не к жизни процесса.

Живой процесс экономит только локальное — спавн процесса, перечитывание транскрипта с диска, переподключение MCP. Это латентность и CPU, не токены.

## Измерение: фазовая сессия против свежей на каждый task

Прогон одной фазы (7 task-ов) с `enable_phase_sessions: true` стоит примерно вдвое больше по расходу 5-часового лимита, чем с `false`.

Причина — не «холодное» перечитывание, а **объём тёплых `cache_read` растущего префикса**. При включённой сквозной сессии планировщик к последнему task тащит в префиксе все предыдущие планы, ревью и tool-выхлопы. Каждый его ход перечитывает весь этот префикс — по дешёвой cache-read ставке, но объём копится сверхлинейно по числу task-ов. При свежей сессии каждый task стартует с маленьким префиксом.

Это та же причина, по которой живой процесс не помогает: он переиспользовал бы тот же растущий префикс по той же ставке.

## Качество

В прогонах с включёнными фазовыми сессиями не зафиксировано выигрыша по качеству или скорости. Два прогона с включённым флагом разошлись между собой по длительности сильнее, чем включённый прогон расходится с выключенным — разброс от недетерминизма модели перекрывает эффект флага. Тестовая фаза состояла из мелких, слабосвязанных task-ов (каждый — отдельный файл по самодостаточной ноте), где межзадачная память почти не нужна.

## Когда включать

Дефолт — `false`: вдвое дешевле, без потери качества на типичных фазах.

`true` имеет смысл рассматривать только для сильно связанных фаз, где решение в раннем task неочевидно ограничивает поздний и важно помнить ход рассуждений, а не только итог. Но даже там дешевле передавать контекст краткой выжимкой через файл: все планы и ревью уже лежат в `.ai-factory/`, и планировщик читает их по запросу. Сессия несёт сырой разговор, файлы несут выводы — для качества обычно достаточно выводов.
