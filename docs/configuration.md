# Конфигурация

## Файл конфигурации

Все настройки оркестратора задаются в `orchestrator.json` в корне репозитория. Файл обязателен — без него оркестратор не запустится. Он gitignored, поэтому изменения настроек не светятся в `git diff`; шаблон для копирования — `orchestrator.json.example`.

```json
{
  "max_iterations": 3,
  "usage_threshold_5h": 90,
  "usage_threshold_weekly": 95,
  "enable_phase_sessions": false
}
```

Все четыре поля обязательны. Отсутствие файла или любого из полей — ошибка при запуске с внятным сообщением.

Путь к файлу можно переопределить через env-переменную:

```bash
ORCHESTRATOR_CONFIG=/path/to/my-config.json uv run orchestrator implement /path/to/project
```

### max_iterations

Лимит итераций для каждого цикла — план-ревью, код-ревью и test-run. Каждый цикл получает до N попыток: если агент не достиг `PASS`-сигнала за N итераций, milestone объявляется несошедшимся — это отказ (см. [Отказы и остановки](failures-and-halts.md)), последнее ревью остаётся на диске.

### usage_threshold_5h

Перед каждым milestone оркестратор запускает `claude /usage` и проверяет два порога независимо. Этот порог — использование текущей 5-часовой сессии. Превышение останавливает прогон до начала следующего milestone.

В лог выводится строка перед каждым milestone:

```
  [usage: session 26% · week 52%]
```

Если парсинг `/usage` не удался — предупреждение в лог, прогон продолжается.

### usage_threshold_weekly

Недельный порог потребления по всем моделям (%). Проверяется из той же строки `claude /usage` независимо от сессионного. Если `claude /usage` не возвращает недельную статистику, порог игнорируется.

### enable_phase_sessions

Когда `false`, оркестратор не передаёт session ID PlannerReviewer между milestone-ами даже внутри одной фазы роадмапа — каждый milestone начинает со свежей сессии. Позволяет сравнить расход токенов на идентичном прогоне с фазовыми сессиями и без.

## Модели агентов

| Агент | Модель | Усилие | Сессия |
|---|---|---|---|
| PlannerReviewer | Opus | high | Персистентная внутри milestone и внутри фазы |
| PlanReviewer | Opus | high | Свежая на каждую попытку |
| Implementer | Sonnet | high | Персистентная внутри milestone |
| TestRunner | — | — | Нет (shell-исполнитель) |

Модели заданы в коде и не конфигурируются через файл настроек.

**PlannerReviewer** держит одну сессию через всю фазу роадмапа (любая группа `##` или `###` заголовков). К третьему milestone фазы планировщик уже знает архитектуру, описание стека и контекст предыдущих планов. На границе фазы сессия сбрасывается. Поведение управляется полем `enable_phase_sessions`.

## Telegram-уведомления

Опциональные поля для отправки уведомлений в Telegram при остановке оркестратора, завершении milestone или окончании всего прогона:

```json
{
  "telegram_bot_token": "1234567890:AAF...",
  "telegram_chat_id": "-1001234567890",
  "telegram_alerts": ["stop", "halt", "milestone", "done"]
}
```

`telegram_bot_token` и `telegram_chat_id` — учётные данные бота. Если любое из двух пустое или отсутствует, уведомления не отправляются (silent no-op, не ошибка).

`telegram_alerts` — список типов событий для уведомлений. Цвет сообщения кодирует исход прогона (концепт — [Отказы и остановки](failures-and-halts.md)):

| Тип | Цвет | Когда отправляется |
|---|---|---|
| `stop` | 🔴 | Отказ — milestone не сошёлся за лимит итераций |
| `halt` | 🟡 | Остановка — исчерпан лимит/внешний ресурс, инфра-сбой или ручная остановка |
| `milestone` | 🟢 | Milestone успешно завершён и закоммичен |
| `done` | 🟢 | Все milestone-ы в роадмапе выполнены |

Пустой список `[]` отключает все уведомления даже при заполненных учётных данных. Сетевая ошибка при отправке не останавливает прогон.
