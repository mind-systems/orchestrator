# Конфигурация

## Файл конфигурации

Все настройки оркестратора задаются в `orchestrator.json` в корне репозитория. Файл обязателен — без него оркестратор не запустится. Он gitignored, поэтому изменения настроек не светятся в `git diff`; шаблон для копирования — `orchestrator.json.example`.

```json
{
  "max_iterations": 3,
  "usage_threshold_5h": 90,
  "usage_threshold_weekly": 95,
  "enable_phase_sessions": false
}
```

Все четыре поля обязательны. Отсутствие файла или любого из полей — ошибка при запуске с внятным сообщением.

Путь к файлу можно переопределить через env-переменную:

```bash
ORCHESTRATOR_CONFIG=/path/to/my-config.json uv run orchestrator implement /path/to/project
```

### max_iterations

Лимит итераций для каждого цикла — план-ревью, код-ревью и test-run. Каждый цикл получает до N попыток: если агент не достиг `PASS`-сигнала за N итераций, task объявляется несошедшимся — это отказ (см. [Отказы и остановки](failures-and-halts.md)), последнее ревью остаётся на диске.

### usage_threshold_5h

Перед каждым task оркестратор запускает `claude /usage` и проверяет два порога независимо. Этот порог — использование текущей 5-часовой сессии. Превышение останавливает прогон до начала следующего task.

В лог выводится строка перед каждым task:

```
  [usage: session 26% · week 52%]
```

Если парсинг `/usage` не удался — предупреждение в лог, прогон продолжается.

### usage_threshold_weekly

Недельный порог потребления по всем моделям (%). Проверяется из той же строки `claude /usage` независимо от сессионного. Если `claude /usage` не возвращает недельную статистику, порог игнорируется.

### enable_phase_sessions

Когда `false`, оркестратор не передаёт session ID PlannerReviewer между task-ами даже внутри одной фазы роадмапа — каждый task начинает со свежей сессии. Позволяет сравнить расход токенов на идентичном прогоне с фазовыми сессиями и без.

### roadmap_path

Необязательное поле — выбирает, какой роадмап читает оркестратор. Три взаимоисключающих состояния:

- **Поле отсутствует или пустая строка** — используется `ROADMAP.md`, поведение не отличается от прогона без этой настройки (byte-stable по умолчанию).
- **Литерал `"my"`** — оркестратор выводит персональный роадмап из git-идентичности этого рабочего места: slug строится из локальной части `git config user.email` (при отсутствии email — из `user.name`), приводится к нижнему регистру, серии не-буквенно-цифровых символов схлопываются в один дефис. Целевой файл — `.ai-factory/roadmaps/<slug>.md`. Если файл существует, его первая строка должна быть `> Owner: <email>`, совпадающим с текущей git-идентичностью — несовпадение или нераспознанная первая строка останавливают прогон операционной остановкой с указанием владельца. Если файл ещё не создан, оркестратор один раз громко сообщает об этом в лог и продолжает на `ROADMAP.md` (ленивая миграция — один и тот же workstation-конфиг работает и до, и после появления именованного роадмапа). Если ни email, ни имя в git не заданы — операционная остановка с подсказкой настроить git-идентичность или указать явный путь.
- **Любое другое значение** — явный относительный путь под `.ai-factory/` (например, `"roadmaps/alice.md"`), используется как есть, без проверки владельца — явное значение выигрывает всегда.

Абсолютный путь или путь с сегментом `..` — ошибка при запуске с указанием некорректного значения.

Тестовый роадмап всегда выводится из основного: для `ROADMAP.md` это `ROADMAP_TESTS.md`, для именованного роадмапа — тот же каталог с суффиксом `-tests` (`roadmaps/alice.md` → `roadmaps/alice-tests.md`). Отдельной настройки для тестового роадмапа нет.

## Оверлей под проект

Глобальный `orchestrator.json` задаёт настройки для всего флота — они применяются к каждому целевому проекту одинаково. Проект переопределяет нужные ему настройки собственным файлом `<проект>/.ai-factory/orchestrator.json`, который накладывается **поверх** глобальной базы.

Приоритет, от низшего к высшему:

1. Значения по умолчанию `OrchestratorConfig` (для необязательных ключей).
2. Глобальный `orchestrator.json` (или путь из `ORCHESTRATOR_CONFIG`) — база.
3. `<проект>/.ai-factory/orchestrator.json` — оверлей проекта.

Оверлей **частичный**: проект указывает только те ключи, которые хочет изменить, — повторять весь набор не нужно. Четыре обязательных ключа базы (`max_iterations`, `usage_threshold_5h`, `usage_threshold_weekly`, `enable_phase_sessions`) в оверлее **не обязательны**: база их уже дала и провалидировала. Ключи, отсутствующие в оверлее, наследуются из базы; присутствующие — побеждают.

Пример — проекту нужно 5 итераций вместо глобальных 3:

```json
{ "max_iterations": 5 }
```

Всё остальное берётся из глобального конфига.

Правила слияния:

- Слияние **поверхностное**: `telegram_alerts` в оверлее **заменяет** список базы целиком, не объединяет.
- `roadmap_path` оверлея проходит тот же guard (относительный путь, без сегмента `..`) и тот же трёхсостоянийный резолвер, что и глобальный.
- Битый JSON оверлея — ошибка при запуске с указанием пути к файлу оверлея.
- **Отсутствие файла оверлея — byte-stable**: для проекта без него поведение не меняется ни на байт.

Оверлей оркестратора живёт в собственном `orchestrator.json` внутри `.ai-factory/` проекта.

## Модели агентов

| Агент | Модель | Усилие | Сессия |
|---|---|---|---|
| PlannerReviewer | Opus | high | Персистентная внутри task и внутри фазы |
| PlanReviewer | Opus | high | Свежая на каждую попытку |
| Implementer | Sonnet | high | Персистентная внутри task |
| TestRunner | — | — | Нет (shell-исполнитель) |

Модели заданы в коде и не конфигурируются через файл настроек.

**PlannerReviewer** держит одну сессию через всю фазу роадмапа (любая группа `##` или `###` заголовков). К третьему task фазы планировщик уже знает архитектуру, описание стека и контекст предыдущих планов. На границе фазы сессия сбрасывается. Поведение управляется полем `enable_phase_sessions`.

## Telegram-уведомления

Опциональные поля для отправки уведомлений в Telegram при остановке оркестратора, завершении task или окончании всего прогона:

```json
{
  "telegram_bot_token": "1234567890:AAF...",
  "telegram_chat_id": "-1001234567890",
  "telegram_alerts": ["task-fail", "stop", "task", "done"]
}
```

`telegram_bot_token` и `telegram_chat_id` — учётные данные бота. Если любое из двух пустое или отсутствует, уведомления не отправляются (silent no-op, не ошибка).

`telegram_alerts` — список типов событий для уведомлений. Цвет сообщения кодирует исход прогона (концепт — [Отказы и остановки](failures-and-halts.md)):

| Тип | Цвет | Когда отправляется |
|---|---|---|
| `task-fail` | 🔴 | Отказ — task не сошёлся за лимит итераций |
| `stop` | 🟡 | Остановка — исчерпан лимит/внешний ресурс, инфра-сбой или ручная остановка |
| `task` | 🟢 | Task успешно завершён и закоммичен |
| `done` | 🟢 | Все task-и в роадмапе выполнены |

Важно при обновлении: литерал `stop` поменял смысл — раньше это был 🔴 отказ, теперь это 🟡 операционная остановка. Автоматической миграции нет: если в `telegram_alerts` уже был `"stop"`, его нужно осознанно пересмотреть под новый набор (например, `["task-fail", "stop", "done"]`).

Пустой список `[]` отключает все уведомления даже при заполненных учётных данных. Сетевая ошибка при отправке не останавливает прогон.
