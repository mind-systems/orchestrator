# Конфигурация

## Лимиты итераций

Оркестратор ограничивает количество циклов implement→review чтобы не зависнуть в бесконечном исправлении. Для обычного режима и режима рефакторинга лимиты настраиваются отдельно через переменные окружения.

`ORCHESTRATOR_MAX_REVIEW_ITERATIONS` — максимум циклов ревью на один milestone в режимах `implement`, `implement-review` и `review`. По умолчанию 3.

`ORCHESTRATOR_MAX_REFACTOR_ITERATIONS` — максимум итераций в режиме `refactor`. По умолчанию 2: первая итерация это аудит и план, вторая — проверка результата.

## Модели агентов

Каждый агент использует свою модель и уровень усилий. PlannerReviewer работает на Opus с высоким усилием — он принимает архитектурные решения и глубоко анализирует код. Implementer работает на Sonnet с высоким усилием — ему нужна скорость при написании кода. PlanReviewer работает на Opus со средним усилием — он делает одиночную проверку плана без длинного контекста.

Модели передаются при создании агентов в `process_milestone()` и `process_refactor_milestone()`. Менять их можно напрямую в коде — конфигурация через env-переменные для моделей не предусмотрена.

## Состояние между запусками

Оркестратор сохраняет своё состояние в `.ai-factory/orchestrator-state.json` в целевом проекте. Сейчас в нём хранится список ревью-файлов, созданных в текущем implement-проходе. Этот файл можно безопасно удалить — оркестратор пересоздаст его при следующем запуске, единственное последствие: перед review-проходом будут удалены все ревью, а не только новые.
