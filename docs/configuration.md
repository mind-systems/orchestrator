# Конфигурация

## Переменные окружения

Все настройки передаются через переменные окружения. Два способа:

```bash
# Однократно для одного запуска
ORCHESTRATOR_MAX_ITERATIONS=5 uv run orchestrator implement /path/to/project

# Для текущей сессии терминала
export ORCHESTRATOR_USAGE_THRESHOLD=85
uv run orchestrator implement /path/to/project
```

| Переменная | По умолчанию | Описание |
|---|---|---|
| `ORCHESTRATOR_MAX_ITERATIONS` | `3` | Лимит итераций для каждого цикла (план-ревью, код-ревью, test-run) |
| `ORCHESTRATOR_USAGE_THRESHOLD` | `90` | Порог использования Claude Code сессии (%), при котором оркестратор останавливается перед следующим milestone |

### ORCHESTRATOR_MAX_ITERATIONS

Один лимит управляет всеми циклами — план-ревью, код-ревью и test-run. Каждый цикл получает до N попыток: если агент не достиг `PASS`-сигнала за N итераций, оркестратор останавливается с содержимым последнего ревью.

Увеличивать имеет смысл для сложных milestone-ов, где один проход план-ревью или код-ревью может не хватить.

### ORCHESTRATOR_USAGE_THRESHOLD

Оркестратор отслеживает использование Claude Code сессии и останавливается до того, как оно достигнет лимита. Проверка адаптивная — не после каждого milestone, а по прогнозу:

1. Проверка перед первым milestone (базовый уровень)
2. Проверка после первого milestone (определяется delta на milestone)
3. Следующая проверка — за один milestone до предсказанной границы: `текущий + ceil((threshold% - current%) / avg_delta) - 1`
4. После каждой проверки прогноз пересчитывается

Для длинного прогона из 20+ milestone-ов оркестратор делает 3-4 проверки вместо 20. Если парсинг `/usage` не удался (сеть, формат изменился) — предупреждение в лог, прогон продолжается.

## Модели агентов

| Агент | Модель | Усилие | Сессия |
|---|---|---|---|
| PlannerReviewer | Opus | high | Персистентная внутри milestone и внутри фазы |
| PlanReviewer | Opus | high | Свежая на каждую попытку |
| Implementer | Sonnet | high | Персистентная внутри milestone |
| TestRunner | — | — | Нет (shell-исполнитель) |

Модели заданы в коде (`process_milestone()`) и не конфигурируются через env.

**PlannerReviewer** держит одну сессию через всю фазу роадмапа (любая группа `##` или `###` заголовков). Это значит, что к третьему milestone фазы планировщик уже знает архитектуру, описание стека и контекст предыдущих планов — не тратит токены на повторное чтение. На границе фазы сессия сбрасывается.
